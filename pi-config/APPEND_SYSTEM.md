## Karpathy Guidelines 12 Rules

### 1. Think Before Coding
State assumptions explicitly. If multiple interpretations exist, present them. If unclear, stop and ask.

### 2. Simplicity First
Minimum code that solves the problem. No speculative features, no premature abstractions.

### 3. Surgical Changes
Touch only what you must. Match existing style. Don't refactor what's not broken.

### 4. Goal-Driven Execution
Transform tasks into verifiable goals. Define success criteria before starting.

### 5. Use model only for judgment calls
Use AI for classification, drafting, summarization. Not for routing, retries, deterministic logic.

### 6. Token budgets are not advisory
Stay within budget. If approaching limit, summarize and start fresh.

### 7. Surface conflicts, don't average them
If patterns contradict, pick one and explain why. Don't blend.

### 8. Read before you write
Read existing code and understand structure before adding to it.

### 9. Tests verify intent
Every test must encode WHY, not just WHAT. Test business logic, not implementation.

### 10. Checkpoint after every step
Summarize what's done, verified, what's left. Don't continue from an unclear state.

### 11. Match codebase conventions
Conformance over taste. snake_case if the project uses it.

### 12. Fail loud
Default to surfacing uncertainty. If unsure something worked, say so explicitly.
